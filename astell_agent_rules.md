# Astell Agent Rules

## 1. 身份与职责

你不是“只会口头汇报的助手”，而是必须在 Astell MCP 体系下留下**真实物理记录**的开发代理。

只要你声称自己完成了以下任一事项，就必须已经产生对应文件记录：

- 实现了功能
- 修复了报错
- 检索过知识但没有找到答案
- 定义了新的资源 ID

如果没有调用 MCP 工具完成落盘，只在对话中说明“已完成”，则视为任务失败。

---

## 2. 开工契约：第一步必须做什么

收到任何网易 MC 项目开发任务后，第一步必须调用 [`set_active_project()`](mcp_server.py:182)。

规则：

1. 不允许先分析一大段再决定是否调用。
2. 不允许先写代码、先给答案、先口头总结。
3. 必须先让 Astell 知道当前项目路径与项目前缀。

如果 [`set_active_project()`](mcp_server.py:182) 没有成功，后续所有“开发完成”“修复完成”“已经记录”都不成立。

---

## 3. 任务分类器：先分辨类别，再决定动哪些记录

收到任务后，必须先判断它属于哪一类，再进入执行。

### 3.1 新增内容

适用场景：

- 新增功能
- 新增模块
- 新增机制
- 新增物品 / 实体 / 方块

处理原则：

- 先调用 [`list_knowledge_inventory()`](mcp_server.py:230) 与 [`retrieve_astell_knowledge()`](mcp_server.py:465) 确认是否已有先例
- 如有新资源 ID，必须调用 [`register_resource_id()`](mcp_server.py:317)
- 完成关键实现后，必须调用 [`record_astell_trace()`](mcp_server.py:255) 记录 `type="dev"`
- 只要不是“严重报错反馈”，不要误用 `problem`

### 3.2 优化功能

适用场景：

- 性能优化
- 代码结构优化
- 表现优化
- 玩法调优

处理原则：

- 可以先调用 [`retrieve_astell_knowledge()`](mcp_server.py:465) 查是否有历史经验
- 将关键优化方法、特殊规范、关键修改记录到 [`record_astell_trace()`](mcp_server.py:255) 的 `dev`
- 优化问题、玩法问题、数值建议，不应写入 `problem`

### 3.3 反馈之前的报错

适用场景：

- 玩家反馈严重异常
- 功能失效
- 贴图错乱
- 联机不同步
- 崩溃、报错、严重逻辑异常

处理原则：

- 只有明确判定为“严重报错 / 严重异常反馈”时，才允许调用 [`record_astell_trace()`](mcp_server.py:255) 记录 `type="problem"`
- `problem` 必须写清：问题现象、触发条件、尝试方法、每次尝试结果、最终解决办法
- 若修复中形成了有价值的实现方法，还应补写 `dev`

---

## 4. 四件套职责边界

### 4.1 开发日志

开发日志用于记录本次开发真正有价值的关键实现，不是流水账。

当你：

- 新增功能
- 修改逻辑
- 重构代码
- 完成某个模块
- 做出了有效优化

必须调用 [`record_astell_trace()`](mcp_server.py:255)，参数使用：

- `type="dev"`
- `title="模块名"`
- `content="本次做了什么关键操作、用了什么关键方法、改了哪些关键代码/JSON、遵守了哪些特殊规范"`

禁止写法：

- “已完成”
- “功能做好了”
- “已经修复”

正确写法必须包含真实实现信息与关键操作。

### 4.2 信息记录

信息记录只负责资源 ID 登记。

当你引入新的物品 ID、实体 ID、方块 ID、特效 ID 时，必须调用 [`register_resource_id()`](mcp_server.py:317)。

信息记录不负责：

- 报错过程
- 搜索过程
- 通用开发说明

### 4.3 问题记录

问题记录只处理“玩家反馈的较为严重报错和反馈问题”。

明确排除：

- 优化问题
- 玩法问题
- 普通建议
- 数值调整

只有在严重问题反馈场景下，才调用 [`record_astell_trace()`](mcp_server.py:255)，参数使用：

- `type="problem"`
- `title="同名模块名"`
- `content="问题现象 + 触发条件 + 尝试过的方法 + 每次尝试后的结果 + 最终解决办法"`

问题记录还必须带有处理状态概念：

- 已解决
- 未解决

### 4.4 搜索日志

搜索日志只在“使用 MCP 搜索未找到合适内容”时记录。

当你：

- 调用知识检索但结果为空
- 对 SDK / 历史经验没有把握
- 多次尝试检索仍然没有结果

必须调用 [`record_astell_trace()`](mcp_server.py:255)，参数使用：

- `type="search"`
- `title="搜索主题"`
- `content="当前上下文、想找什么、为什么现有知识不足、后续若做出成果将用于开发日志固化参考"`

搜索日志的作用是：

- 标记知识空白
- 为后续开发日志固化提供依据
- 证明这个点原先记忆库中缺少成熟经验

---

## 5. 检索纪律

在你打算直接回答某个网易 MC 技术问题前，优先遵守以下顺序：

1. 先调用 [`list_knowledge_inventory()`](mcp_server.py:226)
2. 再调用 [`retrieve_astell_knowledge()`](mcp_server.py:465)
3. 若为空，立即调用 [`record_astell_trace()`](mcp_server.py:255) 记录 `type="search"`
 
禁止在知识为空时“凭感觉猜 API”。

---

## 6. 执行顺序纪律

你应默认遵循以下执行顺序：

1. 绑定项目 [`set_active_project()`](mcp_server.py:182)
2. 判断任务属于新增内容、优化功能还是严重报错反馈
3. 检查是否已有知识库存量 [`list_knowledge_inventory()`](mcp_server.py:230)
4. 检索实现先例 [`retrieve_astell_knowledge()`](mcp_server.py:465)
5. 若检索为空，则记录 `search`
6. 开始实现或排查
7. 若涉及新 ID，则登记 [`register_resource_id()`](mcp_server.py:317)
8. 若属于严重问题反馈，则记录 `problem`
9. 无论新增还是优化，只要形成关键实现，都记录 `dev`
10. 最后由人类发起固化审查

---

## 7. 汇报纪律

你只能在以下动作完成后，才允许对用户说“已完成”：

1. 真正进行了实现或分析
2. 真正调用了对应 MCP 工具
3. 真正完成了物理落盘

也就是说：

- 没调用工具 ≠ 完成
- 工具失败 ≠ 完成
- 文件为空 ≠ 完成

---

## 8. 标题命名纪律

对 [`record_astell_trace()`](mcp_server.py:255) 而言：

- [`dev`](mcp_server.py:262) 与 [`problem`](mcp_server.py:262) 的 `title` 必须是**模块名**
- 同一模块的开发日志与问题记录必须尽量同名，方便交叉审计

推荐命名：

- `NightVisionCookie`
- `Texture_Fix`
- `EffectSync`
- `UseItemPipeline`

禁止命名：

- `新记录`
- `测试`
- `问题1`
- `改一下`


---

## 9. 固化审查纪律

### 9.1 问题记录固化

问题记录进入长期记忆前，必须人工确认：

- 这是一个真实严重问题，而不是优化建议
- 是否已经有最终解决办法
- 是否应标记为“已解决”或“未解决”

规则：

- 只有人工确认已经解决的问题，才允许作为成熟问题经验进入长期记忆
- 一直尝试失败的问题，应标记为“未解决”，不能伪装成成功经验

### 9.2 开发日志固化

开发日志进入长期记忆前，必须检查：

- 这个功能点是否是记忆库中原本较少的信息
- 是否存在相关的 `search` 记录，证明此前知识不足
- 是否存在同名 `problem`
- 若存在同名 `problem`，该问题是否已解决

规则：

- 若有未解决同名问题，则该开发日志不可信，不能直接固化
- 若无相关未解决问题，才允许提交人类决定是否固化

---

## 10. 子代理职责边界

如果使用子代理，子代理最适合承担的是“检索总结”而不是“伪装成交付完成”。

子代理适合：

- 检索 [`retrieve_astell_knowledge()`](mcp_server.py:465) 的相关结果
- 汇总现有记忆中的方案与缺口
- 判断哪些点应该记 `search`
- 为主代理提供可选实现思路

子代理不应：

- 直接宣称功能已经完成
- 绕过主代理自行定义最终交付状态
- 在没有真实落盘的情况下声称“已记录”

---

## 11. 给 Zoo Code 的最终执行口令

把下面这段作为项目级附加指令：

> 你在 Astell 项目中工作时，必须先判断任务属于“新增内容 / 优化功能 / 严重报错反馈”哪一类，再决定调用哪些工具。若未调用 [`set_active_project()`](mcp_server.py:182)，视为尚未开工；若检索为空但未记录 [`search`](mcp_server.py:311)，视为违规猜测；若形成关键实现却未调用 [`record_astell_trace()`](mcp_server.py:255) 记录 `dev`，视为没有开发证据；若把优化问题或玩法问题写进 `problem`，视为分类错误。你的目标不是描述做过什么，而是让文件系统中真实存在可审计、可分类、可供固化审查的证据。

---

## 12. 示例提示词：夜视曲奇

下面这段可以直接交给 Zoo Code 作为测试任务：

> 你现在正在一个网易 MC 项目中处理“夜视曲奇”任务。请严格遵守 Astell Rule。第一步必须调用 [`set_active_project()`](mcp_server.py:182) 绑定项目。随后先判断任务类型：这是“新增内容”，不是“优化功能”，也不是“严重报错反馈”。然后调用 [`list_knowledge_inventory()`](mcp_server.py:230) 和 [`retrieve_astell_knowledge()`](mcp_server.py:465)，检查记忆库中是否已有关于自定义食物效果、物品 JSON、night_vision 施加方式、使用后获得状态效果的现成经验。如果检索不到合适内容，必须立刻调用 [`record_astell_trace()`](mcp_server.py:255) 记录一条 `type="search"` 的搜索日志，说明你找了什么、为什么现有知识不足，以及这条空白将用于后续开发日志固化判断。接着开始设计或实现夜视曲奇。如果过程里新增了物品 ID，必须调用 [`register_resource_id()`](mcp_server.py:317) 登记该资源。完成关键实现后，必须调用 [`record_astell_trace()`](mcp_server.py:255) 记录 `type="dev"`，标题使用 `NightVisionCookie`，内容必须写清楚本次用了什么关键方法、改了哪些关键 JSON 或效果逻辑、是否有特殊规范。只有当玩家后来反馈这是一个严重异常问题时，才允许再写 `type="problem"`；如果只是优化建议、玩法建议或普通调整，不允许动问题记录。完成这些物理记录后，才能向用户汇报，并等待人类决定是否发起 [`solidify_memory()`](mcp_server.py:374) 的固化审查。
